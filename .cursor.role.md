## 2. Component patterns

### 2.1 Layout hierarchy (page structure)

- **Root**: `src/app/_layout.tsx` – Expo Router `Stack` with `headerShown: false`.
- **Screen shell**: Every main screen uses **AppLayout**:
  - `SafeAreaView` (flex 1, background `#191021`)
  - **TopBar** (variant by `type`: `"auth"` | `"unAuth"`)
  - **ScrollView** (flex 1, horizontal padding 16, top padding 8) wrapping **children**
  - **BottomNavigation**

Example (from `app/index.tsx`):

```tsx
import { AppLayout } from "../components/layout";

export default function Index() {
  return (
    <AppLayout>
      <View style={styles.container}>
        <Text>Welcome to AM Arena!</Text>
      </View>
    </AppLayout>
  );
}
```

- **Rule**: New screens that need the app shell must wrap their content in `<AppLayout>{children}</AppLayout>` and keep TopBar + BottomNavigation usage consistent.

### 2.2 Component structure and props

- **Functional components** with `React.FC<Props>` (or explicit props types).
- **Typed props**: Each component declares a `Props` or `type Props` (and optional props like `username?`, `active?`).
- **Styles**: `StyleSheet.create({ ... })` at the bottom of the file; no inline styles for complex UIs except where already used (e.g. AppLayout SafeAreaView/ScrollView).
- **Exports**: Default export for UI components; named exports for layout/public API (e.g. `export { AppLayout }` from `layout/index.ts`, `export { Factory as TopBar }` from `TopBar/index.ts`).
- **Reusability**: Components receive data and callbacks via props (e.g. `BottomNav`: `activeTab`, `onTabPress`; `BottomTabItem`: `label`, `icon`, `active`, `onPress`). New components must be reusable and receive props properly.

### 2.3 Feature folders (TopBar / BottomNav)

- **TopBar**: Factory pattern – `Factory` (exported as `TopBar`) composes `Init` + `UiFactory`. `Init` mirrors `type` into the mirror store; `UiFactory` reads `type` and renders `LoggedInTopBar` or `GuestTopBar`.
- **BottomNav**: Container/presentational split – `BottomNavigation` holds local state (`activeTab`, `setActiveTab`) and passes them to presentational `BottomNav`; `BottomNav` renders a list of `BottomTabItem`s with `label`, `icon`, `active`, `onPress`.

### 2.4 Feature architecture (layered) – standard for new features

All new feature components **must** follow this layered architecture. Reference implementations: **TopBar**, **home/super-sub**, **home/latest-winner**, **home/top-player**, **home/tournament-card**.

**Required folder structure per feature:**

```
feature-name/
├── api/           # API logic when needed (can be empty index.ts)
├── utils/         # Feature helpers when needed (can be empty index.ts)
├── state/         # Feature state (mirror store)
│   ├── init.ts    # InitState type and default store()
│   └── index.ts   # mirrorFactory(InitState()), useMirror, useMirrorRegistry
├── init.tsx       # Receives props, registers into store, renders children only
├── factory.tsx    # Receives props, composes Init + UiFactory
├── ui/
│   ├── ui-factory.tsx   # Reads store via useMirror, renders presentational view(s)
│   └── [View].tsx       # Pure UI – no props from outside; data from store or from UiFactory
└── index.ts       # Export only the factory (e.g. export { Factory as FeatureName })
```

**Layer separation rules (strict):**

1. **Props enter the feature only through Init.** No feature data props may be passed directly to UI or other layers. The parent passes props to the Factory; the Factory passes them to Init; Init registers them into the feature store via `useMirrorRegistry`.
2. **UI layer (ui/) must NOT contain:** API calls, utils logic, or state logic (no direct store writes). UI only renders data. Data is read via `useMirror` in the UI layer or passed from UiFactory (which read from the store) to presentational sub-views.
3. **state, api, and utils** remain isolated from UI; no UI imports inside these layers.
4. **Multi-instance features** (e.g. list items): Store shape is a keyed map (`byId`). The parent builds the full map and passes `byId` + `instanceId` to each Factory. Init registers `byId`; UiFactory receives `instanceId` as a selector and reads `byId[instanceId]` from the store to render the correct card.

**Data flow:** Parent → Factory (props) → Init (registers props into store) → UiFactory (reads store via `useMirror`) → presentational view (reads store or receives slice from UiFactory).

## 3. State management

### 3.1 Zustand and mirror store

- **Zustand** is used via a custom **mirror factory** in `src/hooks/use-mirror-factory.ts`.
- **No global app store**; state is scoped to features (e.g. TopBar has its own store).

### 3.2 Mirror factory (TopBar store)

- `mirrorFactory(store)` creates an internal Zustand store and returns:
  - **useMirrorRegistry(key, value, type?)**: register a value (or ref) into the store; used at the top of the tree (e.g. in `Init`).
  - **useMirror(key)**: read a value from the store; used in descendants (e.g. in `UiFactory`).
- TopBar state is defined in `TopBar/store/init.ts`:
  - **InitState**: `{ type: "auth" | "unAuth" }`
  - Default: `type: "unAuth"`.
- **UI variant**: `type === "auth"` → `LoggedInTopBar`; otherwise → `GuestTopBar`. The `type` is passed as a prop to `<TopBar type="auth" />` (or `"unAuth"`) from `AppLayout` and mirrored into the store so `UiFactory` can pick the correct bar.

### 3.3 Local state

- **BottomNav** uses `useState("Home")` for `activeTab`; no Zustand. Tab press updates local state only (no routing yet).

## 4. Theming and colors

### 4.1 Central palette

- **File**: `src/theme/colors.ts`.
- **Usage**: Import as `import { colors } from "@/src/theme/colors";` in components.

### 4.2 Color tokens

| Token               | Hex       | Usage                          |
|---------------------|-----------|---------------------------------|
| `primaryPurple`     | `#7e0df1` | Primary brand, active states   |
| `neonBlue`          | `#0fb9d9` | Accent, buttons (e.g. Login)   |
| `darkBackground1`   | `#191021` | Main background (TopBar, BottomNav, SafeAreaView) |
| `darkBackground2`   | `#20142c` | Layered backgrounds, borders  |
| `white`             | `#ffffff` | Text on dark                   |
| `grey`              | `#8a8a8a` | Secondary text                 |
| `error`             | `#ff4d4d` | Errors / alerts                |

### 4.3 Usage in components

- TopBar and BottomNav: `backgroundColor: colors.darkBackground1` (or hardcoded `#191021` where already used).
- Active tab text: `colors.primaryPurple`; inactive: `colors.white`.
- Buttons (e.g. Login): `colors.neonBlue` background, `colors.white` text.
- Logo/brand text: `colors.primaryPurple`.
- **Rule**: New components must use `colors.*` (or the same hex values) for consistency; avoid introducing new ad-hoc hex values.

## 5. Icons and assets

### 5.1 Vector icons

- **Library**: `react-native-vector-icons`.
- **Set used**: **MaterialIcons** only (e.g. `home`, `sports-esports`, `movie`, `video-library`, `person`, `notifications`, `currency-bitcoin`).
- **Import**: `import Icon from "react-native-vector-icons/MaterialIcons";`
- **Usage**: `<Icon name="…" size={…} color="…" />` (size and color as props). For bottom tabs, icon is passed as `React.ReactNode` into `BottomTabItem`.
- **Rule**: Use MaterialIcons (or the same library’s other sets if needed) for standard UI icons; reserve SVGs for custom/Neon/brand assets.

### 5.2 Images and SVGs

- **Images**: `src/assets/main_Logo.png` – used in `LoggedInTopBar` via `require("../../../assets/main_Logo.png")`.
- **SVGs**: No SVG usage in `src` yet. Use SVGs only for custom/Neon/branding icons when needed.
- **Rule**: Standard icons → vector-icons; custom/brand visuals → assets (PNG/SVG).

## 6. Navigation and routing

### 6.1 Expo Router

- **Entry**: `app/_layout.tsx` – root `Stack`, `screenOptions={{ headerShown: false }}`.
- **Index**: `app/index.tsx` – single screen wrapping content in `AppLayout`.
- **No file-based routes** beyond index yet; new screens will be added under `app/` (e.g. `app/(tabs)/…` or `app/screen-name.tsx`) and should follow the same AppLayout pattern where the full shell is required.

### 6.2 Wrappers around page content

- **AppLayout**: Wraps every main screen content (SafeAreaView → TopBar → ScrollView(children) → BottomNavigation).
- **TopBar**: Rendered inside AppLayout; variant by `type` prop (`"auth"` | `"unAuth"`).
- **BottomNavigation**: Rendered inside AppLayout; currently local state only, no route linking.

## 7. Rules for new work

1. **Feature architecture**: New features **must** use the layered structure (api/, utils/, state/, ui/, init.tsx, factory.tsx, index.ts). Props enter only through Init; UI reads only from the store (or from UiFactory that read from the store). See **2.4 Feature architecture (layered)**.
2. **Structure**: Place new features under `components/` (e.g. `components/home/feature-name/`); new screens under `app/`; theme in `theme/`; shared hooks in `hooks/`.
3. **Layout**: Use **AppLayout + TopBar + BottomNavigation** for any new main screen that should have the full app shell.
4. **Components**: Build reusable components with explicit **Props** types; for features, props go to Factory/Init only; UI is pure and data-driven from the store.
5. **Theme**: Use **`colors`** from `@/src/theme/colors` for all background, text, and accent colors; do not introduce new arbitrary hex values.
6. **Icons**: Use **react-native-vector-icons** (MaterialIcons) for standard icons; use **SVGs/images** only for custom or Neon/brand assets.
7. **State**: Use the **mirror factory** pattern (Zustand) for feature state; use local `useState` only for non-feature component state.
8. **Imports**: Use `@/src/…` for theme and shared modules; relative imports for same-feature or nearby components.
9. **No modifications**: This document is for reference and for guiding **new** components, pages, and layouts; do not change existing code when applying this role.
